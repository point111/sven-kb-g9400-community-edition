# Простой рабочий процесс

Этот проект ведётся одним владельцем и не требует сложной Git-модели.

## Ветки

- `main` — только проверенные версии и релизы.
- `develop` — текущая версия, которую готовим к следующему тестированию.
- `feature/...` — временные ветки только для крупных отдельных изменений.

## Обычный цикл

1. Новые изменения добавляются в `develop` или отдельную `feature/...` ветку.
2. Открывается Pull Request в `develop`.
3. Перед merge проверяется, содержит ли PR новое принятое проектное решение. Если содержит, это решение добавляется в `docs/CHAT_HISTORY.md` в том же PR.
4. После проверки изменения объединяются с `develop`.
5. Когда сборка готова к тестированию, открывается Pull Request `develop` → `main`.
6. После merge в `main` создаётся GitHub Release.

## Документы проекта

- `CHANGELOG.md` — что изменилось в версиях проекта.
- `ROADMAP.md` — что планируется реализовать.
- `docs/CHAT_HISTORY.md` — только принятые решения, причины и последствия.

`docs/CHAT_HISTORY.md` не должен превращаться в копию переписки. Исправления опечаток и другие изменения без нового проектного решения отдельной записи не требуют.

## Что нажимать владельцу проекта

### Принять изменение

1. Открыть Pull Request.
2. Проверить вкладки **Files changed** и **Checks**.
3. Проверить, зафиксированы ли новые принятые решения в `docs/CHAT_HISTORY.md`.
4. Нажать **Merge pull request**.
5. Нажать **Confirm merge**.
6. При необходимости нажать **Delete branch** для временной ветки.

### Выпустить тестовую версию

1. Убедиться, что нужные изменения находятся в `main`.
2. Открыть раздел **Releases**.
3. Нажать **Draft a new release**.
4. Создать тег вида `v1.0.0-alpha1`.
5. Прикрепить установщик и контрольную сумму SHA-256.
6. Отметить релиз как **pre-release**.

## Правило безопасности

В репозиторий не добавляются оригинальные EXE, MSI, ZIP и фирменные ресурсы SVEN. Проект хранит только собственный код, документацию и данные, необходимые для применения изменений к официально установленному ПО.
