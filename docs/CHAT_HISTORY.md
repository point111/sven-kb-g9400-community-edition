# История принятых решений

Этот файл хранит только решения, которые были приняты и повлияли на проект. Он не является копией переписки и не содержит отклонённые варианты, промежуточные идеи или необязательные рассуждения.

## Правило ведения

- При подготовке каждого содержательного Pull Request проверяется, появились ли новые принятые решения.
- Если решение принято, оно добавляется сюда в том же Pull Request.
- Запись объясняет само решение, его причину и последствия для проекта.
- Исправления опечаток и другие изменения без нового проектного решения отдельной записи не требуют.

## 2026-07-16 — PR #1: начальная структура Community Edition

### Решение

Создать независимый проект **SVEN KB-G9400 Community Edition** в виде патча к официально установленному конфигуратору, а не распространять изменённый оригинальный EXE, MSI, архивы или фирменные ресурсы SVEN.

Патч должен:

- проверять ожидаемое состояние оригинального файла перед изменением;
- создавать резервную копию;
- позволять полностью восстановить оригинал;
- сохранять штатный HID-протокол и функции конфигуратора.

### Причина

Ошибка `Product and driver are inconsistent` вызвана дополнительной проверкой идентификационной строки устройства, хотя конфигуратор поддерживает HID-интерфейс исследованной клавиатуры. Распространение только собственного кода и данных патча безопаснее и не требует включать в репозиторий чужие бинарные файлы.

### Последствия

Репозиторий хранит только собственные скрипты, документацию и данные для применения изменений к отдельно установленному официальному ПО. Начальная совместимость подтверждена на Windows 11 25H2 и устройстве `VID_30FA&PID_2350`.

## 2026-07-16 — PR #2: рабочий процесс проекта

### Решение

Использовать облегчённую схему разработки:

- `main` — проверенные версии и релизы;
- `develop` — следующая тестовая версия;
- временные `feature/...`-ветки — только для отдельных крупных изменений;
- перенос готовой версии из `develop` в `main` выполняется через Pull Request.

Также приняты `ROADMAP.md` и `CHANGELOG.md` как постоянные документы проекта.

### Причина

Проект ведётся одним владельцем и не нуждается в сложной корпоративной Git-модели, но изменения должны оставаться проверяемыми и воспроизводимыми.

### Последствия

Новая разработка ведётся в `develop` или временной ветке, а `main` используется как стабильная точка проекта.

## 2026-07-16 — PR #3: подсветка по языку ввода

### Решение

Добавить автоматическое изменение цвета или профиля подсветки при переключении активной раскладки Windows в раздел экспериментальных функций.

Функция не входит в обязательный состав v1.0.0 и не должна задерживать базовый релиз. Реализация допускается только после подтверждения безопасной HID-команды и отсутствия конфликтов со штатным конфигуратором.

### Причина

Функция потенциально полезна пользователям, но требует отдельного фонового приложения, надёжного отслеживания раскладки и дополнительного исследования HID-протокола.

### Последствия

Исследование ведётся отдельно от основной линии выпуска. До подтверждения безопасности функция остаётся экспериментальной и не обещается пользователям как часть стабильной версии.

## 2026-07-16 — PR #5: безопасность alpha1 и источник локализации

### Решение

Считать `localization/ru-RU.csv` единым проверяемым источником русской терминологии. Не публиковать готовый языковой payload до появления воспроизводимого процесса извлечения и сборки ресурсов без включения оригинальных файлов SVEN.

Усилить патчер обязательной проверкой резервной копии и результата по SHA-256, атомарной записью EXE и диагностическим журналом.

### Причина

Текст перевода должен быть отделён от фирменных бинарных и графических ресурсов. Изменение установленного EXE должно либо завершаться проверенным результатом, либо останавливаться без изменения файла.

### Последствия

Русская терминология может независимо проверяться и дополняться в репозитории. RU-payload остаётся незавершённой частью локализации. Обновлённый патчер должен быть повторно протестирован владельцем на подтверждённой конфигурации до переноса в `main`.

## 2026-07-16 — PR #10: граница версий 1.x и 2.x

### Решение

Выпустить `v1.0.0` как стабильный патч совместимости фирменного конфигуратора с Windows 11. Версия 1.x не добавляет функций, которых нет в оригинальном ПО, и не включает локализацию интерфейса.

Все дополнительные возможности перенести в версию 2.x. Поскольку оригинальный EXE не предоставляет интерфейс языковых плагинов, локализация в 2.x может использовать либо воспроизводимое патчинговое изменение ресурсов проверенной версии EXE, либо отдельный загрузчик/компонент, который определяет окружение и подключает только совместимые расширения.

### Причина

Патч совместимости уже полностью проверен и готов к использованию. Локализация и новые возможности имеют другой риск и жизненный цикл и не должны смешиваться с надёжной веткой совместимости.

### Последствия

Ветка 1.x остаётся стабильной и обслуживает только совместимость. Ветка 2.x допускает контролируемое изменение ресурсов или подключение отдельных компонентов, но репозиторий по-прежнему не распространяет оригинальные или заранее модифицированные бинарные файлы SVEN.

## 2026-07-16 — PR #11: графический установщик и деинсталлятор

### Решение

Публиковать стабильный патч 1.x в виде обычного графического Windows-инсталлятора, собранного Inno Setup, и регистрировать штатный деинсталлятор в Windows Installed apps.

Инсталлятор остаётся оболочкой над проверенными скриптами проекта: он не содержит оригинальный или заранее пропатченный EXE, проверяет наличие официального ПО, применяет патч и запускает автоматическую проверку. Деинсталлятор обязан восстановить оригинальный EXE до удаления инструментов восстановления.

Первый выпуск инсталлятора публикуется без цифровой подписи с обязательной SHA-256 контрольной суммой.

### Причина

Рабочий патч должен быть доступен пользователям без навыков PowerShell и Git. При этом упрощение установки не должно ослаблять проверку версии, резервное копирование и гарантированный откат.

### Последствия

Основной пользовательский способ установки — `Setup.exe`; ручные PowerShell-скрипты сохраняются для разработки и диагностики. Перед публикацией установщик и деинсталлятор должны пройти полный тест на реальной системе.
